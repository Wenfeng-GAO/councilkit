# CouncilKit - 产品需求文档

## 产品目标

帮助个人用户通过多 AI agent 的结构化讨论来提升决策质量。解决的核心问题是：当前用户需要在多个模型（Claude、DeepSeek、GPT 等）之间手动切换、重复提问、自己综合结论，过程低效且容易遗漏好观点。CouncilKit 把"多模型多角色讨论"变成一个可操作的产品——创建房间、定义 agent、发起讨论、自动生成结论。

## 目标用户

### 主要用户

- 画像: 独立开发者/知识工作者，日常需要做各类决策和创意讨论
- 场景: 设计技术方案、给项目命名、讨论社会现象、对比选型方案等需要多角度思考的场景
- 动机: 通过观点碰撞提升决策质量，而非仅靠单一模型的视角
- 当前替代方案: 在 Claude、DeepSeek、GPT 等多个 tab 之间手动切换提问，自己在脑中综合结论。代价是重复劳动、容易遗漏观点、缺少真正的观点碰撞

### 次要用户

本阶段不定义次要用户。产品先解决个人使用场景，验证核心价值后再考虑团队协作。

## 核心场景与用户旅程

### 场景 1: 快速发起一次多 agent 讨论

- 触发: 用户遇到一个需要多角度思考的问题（例如"给新项目起个名字"）
- 过程: 打开 CouncilKit → 新建房间 → 输入讨论话题 → 选择/创建 agent（选模型、可选角色定义）→ 发起讨论
- 反馈: agent 依次或并行发言，用户实时看到讨论过程，可随时插入自己的观点参与讨论
- 完成: 讨论结束后自动生成总结/决策摘要，用户获得多角度综合结论
- 异常/边界: 某个模型 API 超时或不可用时，其他 agent 继续讨论并标注该 agent 离线；话题为空时提示输入

### 场景 2: 在已有房间中继续深入讨论

- 触发: 用户看完上一轮讨论结果，想追问或深入某个方向
- 过程: 打开已有房间 → 看到历史讨论和总结 → 输入追问或新指令 → agent 基于上下文继续讨论
- 反馈: agent 引用之前的讨论内容，给出更深入的回答
- 完成: 更新总结摘要
- 异常/边界: 上下文过长时自动压缩历史；房间长时间未使用时保持可访问但不占活跃资源

## 功能需求

### P0: MVP 必做

- R1: 用户能创建一个讨论房间，指定话题
- R2: 用户能在房间中添加 agent，选择底层模型（至少支持 Claude、DeepSeek、GPT）
- R3: 用户能为 agent 定义角色/立场（例如"产品经理"、"反对者"、"乐观主义者"）
- R4: 发起讨论后，agent 能看到彼此的发言并互相质疑和补充
- R5: 讨论结束后自动生成总结/决策摘要
- R6: 用户能在同一房间中基于当前话题继续追问和讨论
- R7: 响应速度可接受——从发起讨论到看到第一条 agent 发言不超过 10 秒
- R8: 用户能作为参与者在讨论中发言，agent 能看到并回应用户的发言

### P1: 应该做

- R9: 用户能保存和复用 agent 配置（例如常用的"技术评审团"组合）
- R10: 支持独立回答模式——agent 各自独立回答，用户对比结果
- R11: 用户能在讨论中途加入新 agent 或移除 agent
- R12: 讨论历史持久化，可随时回看

### P2: 可以做

- R13: 支持导出讨论记录和总结（Markdown/PDF）

## 成功标准

- SC-1: 用户从打开 app 到发起第一次讨论的操作时间不超过 30 秒
- SC-2: 讨论总结能被用户接受为"有用的综合"（自评，非敷衍的"两边都有道理"）
- SC-3: 用户在需要多角度思考的场景下，选择使用 CouncilKit 而非手动切 tab（周活使用 >= 3 次）
- SC-4: 第一条 agent 发言延迟不超过 10 秒

## 不做什么

| 不做的事项 | 原因 | 何时重新考虑 |
|------------|------|--------------|
| 多人协作（多个真人共享一个房间） | 当前是个人工具，协作增加复杂度 | 用户验证后有明确需求时 |
| 自动选择最佳模型/角色 | MVP 先让用户手动控制，理解用户偏好后再做推荐 | 积累足够使用数据后 |
| 模型微调/训练 | 超出产品范围，使用现有模型 API 能力 | 不考虑 |
| 工作流自动化（类似 AutoGen） | CouncilKit 是讨论工具，不是任务编排框架 | 不考虑 |
| 语音/视频形式的讨论 | 文本讨论是核心，多模态增加复杂度 | 技术成熟且用户有需求时 |

## 风险与未决问题

### 阻断性问题: Resolve Before Design

- 无。核心产品方向已清晰。

### 延后到 Design

- 讨论过程的 UI 呈现：是类似聊天界面的时间线，还是分栏对比，还是其他形式？
- "独立回答"和"互相讨论"两种模式在界面上如何切换？
- 总结摘要的呈现位置和更新时机

### 延后到 Tech

- 多模型 API 的并发调用和延迟控制策略
- 长对话上下文管理（压缩/截断策略）
- 本地数据存储方案（房间和讨论历史）
- Web app vs macOS native app 技术选型

## 假设

- A1: 多个 agent 互相看到对方发言并质疑/补充，能产生比独立回答更高质量的综合结论
- A2: 用户愿意花 30 秒设置 agent（选模型+定角色）来换取更好的讨论质量
- A3: 10 秒内的首条发言延迟对用户来说是可接受的
- A4: 自动生成的总结能有效减少用户自行综合的认知负担

## 术语表

- 房间: 一个绑定话题的讨论空间，包含多个 agent 和讨论历史
- Agent: 房间中的一个 AI 参与者，由底层模型 + 可选角色/立场定义
- 讨论: agent 围绕话题发言、互相回应的过程
- 总结: 讨论结束后自动生成的综合结论/决策摘要
