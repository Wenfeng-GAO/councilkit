# CouncilKit Phase 2 Design Notes

## 1. PRD 解读与设计需求提取

### P0 需求的设计含义

| 需求 | 设计含义 |
|------|----------|
| R1: 创建房间，指定话题 | 需要"新建房间"入口 + 话题输入界面。SC-1 要求 30 秒内完成，流程不能超过 2-3 步 |
| R2: 添加 agent，选择模型 | 需要 agent 配置面板，至少列出 Claude/DeepSeek/GPT 三个模型供选择 |
| R3: 为 agent 定义角色/立场 | agent 配置需要角色输入字段（文本输入或预设选项） |
| R4: agent 互相看到发言并质疑补充 | 核心讨论界面——需要展示多 agent 的对话流，发言有明确归属标识 |
| R5: 讨论结束后自动生成总结 | 需要总结展示区域，位置和时机是设计决策点 |
| R6: 基于当前话题继续追问 | 房间是持久的，需要历史讨论回看 + 追问入口 |
| R7: 首条发言 ≤10 秒 | 需要讨论进行中的 loading 状态反馈（打字指示器或流式输出） |
| R8: 用户作为参与者发言 | 用户输入框需要在讨论进行中仍然可用，用户身份需要和 agent 区分 |

### P1 需求的设计含义

| 需求 | 设计含义 |
|------|----------|
| R9: 保存/复用 agent 配置 | 需要"agent 模板"管理入口，创建房间时能选已有模板 |
| R10: 独立回答模式 | 界面需要支持两种展示模式：对话时间线 vs 分栏对比 |
| R11: 中途加入/移除 agent | 讨论进行中的 agent 管理操作 |
| R12: 讨论历史持久化 | 房间列表/历史回看页面 |

### PRD "延后到 Design" 的问题

1. **讨论过程的 UI 呈现**：时间线 vs 分栏对比 vs 其他形式？
2. **"独立回答"和"互相讨论"两种模式在界面上如何切换？**
3. **总结摘要的呈现位置和更新时机**

### 成功标准对设计的约束

- SC-1（30 秒完成首次讨论发起）→ 创建房间+配置 agent+发起讨论的总流程步骤不能超过 3 步
- SC-4（首条发言 ≤10 秒）→ 必须有明确的"讨论已开始"反馈，避免用户以为卡死

### 假设对设计的影响

- A2（用户愿意花 30 秒设置 agent）→ agent 配置不能太复杂，但用户有意愿做基础设置
- A4（总结能减少认知负担）→ 总结不能藏得太深，需要在讨论结束时自然呈现

## 2. 竞品 UI 模式研究

### 同类产品分析

| 产品 | 页面结构 | 核心交互模式 | 视觉调性 |
|------|----------|--------------|----------|
| MindStudio Agent Chat Rooms | 对话式时间线，多 agent 发言以不同头像/名称区分 | 聊天界面+角色面板 | 简洁工具风 |
| TypingMind | 侧边栏对话列表+主内容区对话 | 标准聊天 UI，支持多模型切换 | 深色/浅色主题，信息密集 |
| ChatGPT (多模型对比) | 分栏对比布局 | 同一问题多模型回答并排对比 | 干净简约 |
| Discord/Slack | 频道列表+消息时间线 | 群聊模式，每条消息有作者标识 | 紧凑信息流 |

### 核心交互范式总结

**方案 A: 聊天时间线**（类 Discord/Slack 群聊）
- 所有发言按时间顺序排列，每条消息标注发言者（agent 或用户）
- 适合"互相讨论"模式，对话流自然
- 信息密度中等，阅读体验线性

**方案 B: 分栏对比**（类 ChatGPT 多模型对比）
- 每个 agent 一列，并排展示各自的回答
- 适合"独立回答"模式，方便对比
- 不适合互相引用和质疑的对话

**方案 C: 混合模式**
- 默认时间线展示讨论过程
- 可切换到分栏视图做结果对比
- 复杂度更高，但覆盖两种需求

### 导航模式

同类产品普遍采用**侧边栏+主内容区**：
- 左侧：房间列表/历史
- 中间/右侧：当前讨论内容
- 可选右侧面板：agent 配置/总结

## 3. 设计约束与风险

### 响应式挑战

- PRD 未明确是否需要移动端。产品定位是"独立开发者/知识工作者"的桌面工具，优先桌面体验
- 如果是 Web app，平板适配是合理需求

### 性能对设计的限制

- 多 agent 并行发言时，需要流式展示（逐字出现），否则等待全部完成再展示会违反 SC-4
- 长讨论可能产生大量消息，需要虚拟滚动或分页策略

### 设计歧义点

- "讨论结束"的定义不明确——是 agent 全部发完一轮算结束？还是用户主动点"结束"？还是设定讨论轮数？
- agent 发言顺序：依次顺序发言？还是并行发言全部流式展示？
- 用户参与发言时，如何不打断 agent 的讨论节奏？

### 可访问性

- 多 agent 发言需要明确的视觉区分（不能只靠颜色）
- 讨论流需要支持键盘导航

## 4. Step 2 设计对话:已确认决策

| 决策点 | 用户确认结果 | 解决的延后问题 |
|--------|--------------|----------------|
| 核心讨论 UI 范式 | 聊天时间线为主（方案 A） | 延后问题 1:讨论过程 UI 呈现 |
| 独立回答模式呈现 | 分栏对比作为可切换的次要视图 | 延后问题 2:两种模式如何切换 |
| 讨论结束定义 | 轮数驱动:发起时设定轮数,走完自动停下并生成总结,追问触发新一轮 | 设计歧义点:讨论结束定义 |
| 总结触发时机 | 每轮结束后自动生成 | 延后问题 3:总结呈现位置和更新时机 |
| 同轮 agent 发言顺序 | 依次顺序发言,后者看到前者并回应 | 设计歧义点:发言顺序 |
| 整体视觉调性 | 深色工具风,参考 Linear / TypingMind,信息密集,等宽字体用于引用 | 设计约束 |
| 导航模式 | 侧边栏（房间列表）+ 主内容区 | 信息架构 |

### 推导出的设计含义

- 时间线 + 分栏切换 → 主讨论区需要一个视图切换控件（时间线 / 分栏对比）
- 轮数驱动 → 发起讨论界面需要"轮数"设置项;讨论区需要轮次分隔与进度指示
- 每轮后总结 → 总结以可折叠区块嵌入时间线,每轮一个,最新一轮默认展开
- 依次发言 + SC-4（首条 ≤10 秒）→ 首个 agent 流式输出,打字指示器标识"谁正在发言"
- 深色工具风 → 配色与间距基准需量化(见 DESIGN.md 视觉方向)
